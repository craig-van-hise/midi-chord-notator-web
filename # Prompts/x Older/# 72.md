## 1. Project Context & Objectives
The previous "Phantom Hitbox" architecture caused a complete regression. The absolute-positioned DOM elements were either swallowed by z-index layers or lost their positioning context, resulting in zero selection capability. The objective of this sprint is to completely abandon DOM-based hit-testing. We will implement **Headless Mathematical Hit-Testing**. The canvas container will intercept raw `x/y` mouse coordinates, and we will calculate collisions purely in memory using the exact same layout math used to position the notes.

## 2. Technical Decisions & Dependencies
* **Total DOM Decoupling:** Remove all `data-note-id`, `onClick`, and Phantom Hitbox `<div>` elements. The visual SMuFL notes must have `pointer-events: none` to ensure the parent container receives all pointer events.
* **Math-Based Collision:** On click, use `getBoundingClientRect()` on the main notation container to get the pointer's relative `x/y`. Iterate through the `activeNotes` array, run their pitch through your `notationMath.ts` (or equivalent Y-coordinate function), and check if the pointer's `y` is within a `┬▒ var(--staff-space)` threshold of the calculated note `y`.
* **State Management:** Selection state remains in a mutable Ref (`selectedNoteIds.current`) to prevent re-render loops, with imperative CSS class toggling on the visual elements (or via standard re-render if the canvas is fast enough, but stick to imperative if that was working).
* **Environment Reality Check:** Do NOT trust JSDOM for CSS layout tests. Rely on pure mathematical unit tests for the intersection logic.
