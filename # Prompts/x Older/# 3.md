## 1. Project Context & Objectives
The current rendering engine in `NotationCanvas.tsx` mathematically calculates note Y-coordinates linearly from Middle C. However, the physical CSS positioning of the Treble and Bass staves introduces a visual discontinuity (the gap between staves is physically larger than the musical interval it represents). This causes bass clef notes to render on incorrect lines (e.g., F3 rendering as D3). 

We are implementing a "Split Point" architectural pattern. A new global setting will dictate whether a note belongs to the right hand (Treble grouping) or left hand (Bass grouping). The rendering engine will then use independent anchor points and offset formulas for each group, bypassing the visual gap issue.

## 2. Technical Decisions & Dependencies
* **Files to Modify:** `src/midi/MIDIProvider.tsx`, `src/App.tsx`, `src/components/NotationCanvas.tsx`.
* **State Management:** The `splitPoint` must be stored in the existing `MIDIProvider` context to avoid prop-drilling. Default value: `60` (Middle C4).
* **Settings UI:** A headless modal approach for the settings cog. We will use native HTML `<dialog>` or absolute positioning for a lightweight modal.
* **Mathematical Discontinuity Fix:** * Notes `>= splitPoint` (Treble Group) will calculate their `y` relative to a Treble Anchor.
  * Notes `< splitPoint` (Bass Group) will calculate their `y` relative to a Bass Anchor, applying a hard mathematical shift to jump the visual gap.
