## 1. Project Context & Objectives
The standalone virtual `Keyboard.tsx` requires two major professional upgrades to make it a more viable MIDI controller:
1. **Vertical Velocity Mapping:** When clicking a key, the generated MIDI output velocity (0-127) should be derived from the vertical Y-coordinate of the pointer event. Clicking lower (closer to the player) yields high velocity; clicking higher (closer to the fallboard) yields low velocity.
2. **Hold Mode Rewrite:** The current "Hold Mode" is purely a visual UI trick inside `Keyboard.tsx` and does not actually hold MIDI data sent to external synths. Furthermore, "Hold" and "Toggle" modes are mutually exclusive. We need to implement a true MIDI Hold mode that defers Note Offs until a *new* Note On occurs (when no other physical keys are pressed) or when Hold Mode is disabled.

## 2. Technical Decisions & Dependencies
* **Velocity Calculation:** We will use `e.nativeEvent.offsetY` to get the vertical coordinate of the click relative to the key's bounding box. The velocity formula will be `Math.floor((offsetY / clientHeight) * 127)`. We must clamp this between `1` and `127` (velocity 0 is functionally a Note Off).
* **Exclusive Modes:** In `Keyboard.tsx`, setting Toggle Mode `true` must force Hold Mode `false`, and vice versa.
* **True MIDI Hold Architecture:** * We will add `isHoldModeActive` to the `MidiContext` (similar to `isSustainActive`). 
  * We will track `heldModePendingNoteOffs` inside `MIDIProvider`.
  * When `isHoldModeActive` is true, Note Offs are intercepted and stored in `heldModePendingNoteOffs`.
  * When a Note On arrives, *if* `physicallyHeldNotes.size === 0` (meaning this is the first note of a new chord/phrase), we must flush all notes in `heldModePendingNoteOffs`, sending their Note Offs out to the world, before processing the new Note On.
  * When `isHoldModeActive` is toggled off, we flush the `heldModePendingNoteOffs` queue.
