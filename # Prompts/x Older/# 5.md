# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The application now successfully splits MIDI input across the Grand Staff staves and aligns notes and ledger lines to the correct pitches. However, two high-priority issues remain:
1.  **Rendering Artifact:** Ledger lines are not perfectly bisecting the note heads; they are sitting ~2 pixels too low.
2.  **Missing Notation Logic (Collisions):** When notes in a chord form the interval of a diatonic second (e.g., E and F), their note heads occupy adjacent line/space positions. Without stems to define chirality, these "seconds" must be offset horizontally to avoid overlap. Standard whole-note collision rules apply (often called the "zipper pattern" in clusters).

This phase must correct the rendering alignment and implement an algorithm in `NotationCanvas.tsx` to handle note head collisions for adjacent diatonic pitches.

## 2. Technical Decisions & Dependencies
* **Target Files:** `src/components/NotationCanvas.tsx`.
* **Alignment Fix:** Apply a `-2px` vertical shift (move UP) to the ledger line `<div>` generation logic.
* **Collision Detection Variable:** We must use `stepOffset` (representing the diatonic index from Middle C) to determine adjacency, not MIDI note number (chromatic). Adjacency is defined as: `Math.abs(noteA.stepOffset - noteB.stepOffset) === 1`.
* **Collision Algorithm (Zipper Pattern):** Since the app renders notes imperatively into an active chord system:
    1.  Whenever a new `Note On` event occurs, the entire set of currently active notes within that specific staff group (Treble or Bass) must be sorted by `stepOffset`.
    2.  Iterate through the sorted notes. Identify contiguous clusters of seconds.
    3.  Within each cluster, the lowest note retains its original X position (`translateX: '-50%'`).
    4.  Every subsequent note UP in that contiguous cluster receives an alternating horizontal offset (move right, return left, move right, etc.).
* **Offset Constant:** The standard notehead width in Bravura is roughly `1.18em`. We will define `Note_Offset_X = 1.2 * var(--staff-space)` to provide the necessary horizontal shift.

### Phase 3: Imperative X-Offset Rendering (The Zipper Pattern) — ✅ Completed
* **Objective:** Direct the imperative renderer to apply horizontal `translateX` styles to the calculated clusters.
* **Instructions (Target: `src/components/NotationCanvas.tsx`):**
    1.  **Define X Offset:** `const NSU_PX = parseFloat(getComputedStyle(document.documentElement).getPropertyValue('--staff-space')) || 12; const NOTE_OFFSET_X_PX = NSU_PX * 1.2;`
    2.  Update the `applyNoteheadOffsets` function to process the generated clusters:
        * Iterate through each cluster.
        * Iterate through notes *within* the cluster (index `i`).
        * Apply `style.left` to the DOM element (`lineContainerRef.current[note.id]`):
            * If `i % 2 === 0` (even index in sorted cluster, including lowest): `left: 50%` (original center).
            * If `i % 2 !== 0` (odd index): `left: calc(50% + ${NOTE_OFFSET_X_PX}px)` (offset right).
    3.  **Reset Logic:** Ensure any note NOT in a contiguous cluster (e.g., a single note, or notes in a third/fifth) has its `left` style reset to `50%`.
    4.  Integrate this function call at the end of both `Note On` and `Note Off` processing to ensure collision recalculation occurs whenever the chord changes.
