# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The previous implementation phase suffered from severe UI regressions and logical oversights. The application title is massive, wrapping onto multiple lines, and displaying the wrong text. Header action buttons were injected into the wrong DOM nodes. The MIDI port loopback prevention failed due to a misunderstanding of Web MIDI port ID assignment. Finally, the user has requested a functional pivot: the UI "Sustain" button must be converted into a "Toggle Mode" latching system for the virtual keyboard, while preserving the standard CC 64 sustain behavior for external hardware.

## 2. Technical Decisions & Dependencies
* **Web MIDI API Names:** Input and Output ports for the same hardware device have distinct `id` properties. To prevent loopbacks, the `<select>` disabled check must compare the `port.name` string, not the `port.id`.
* **Title DOM Enforcement:** The header typography must be strictly single-line, using `whitespace-nowrap` and a uniform font size.
* **Toggle Mode Architecture:** * "Toggle Mode" is independent of "Sustain Mode".
  * When Toggle Mode is ACTIVE: Virtual keyboard clicks trigger a Note On. Releasing the mouse (`onPointerUp`/`onPointerLeave`) does NOTHING. Clicking the active key again triggers the Note Off.
  * Hardware CC 64 sustain functionality remains exactly as programmed previously.
