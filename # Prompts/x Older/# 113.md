# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
You are tasked with integrating a new "Transformations" drawer UI into the existing React 19 MIDI Notator application. The drawer must slide out from beneath the `Keyboard.tsx` component. The toolbar controls require deterministic routing to the history and transformation engines located inside `NotationCanvas.tsx`. Furthermore, you must architect a global "Pre-Flight MIDI Interceptor" within `MIDIProvider.tsx` to facilitate the "MIDI Learn" feature, ensuring mapped hardware triggers execute UI actions while being entirely blocked from the notation and audio pipeline.

## 2. Technical Decisions & Dependencies
* **State Decoupling (Event Bus):** Because the Notation/History engine operates via an imperative `useRef` architecture inside `NotationCanvas.tsx`, you will **NOT** lift state to a Context. Instead, you will establish a strict `CustomEvent` bridge (e.g., `APP_TRANSFORM`, `APP_HISTORY`) that the Toolbar dispatches and the Canvas listens to.
* **MIDI Pre-Flight Interceptor:** You must modify `MIDIProvider.tsx` to support a registered interceptor function `(msg: Uint8Array, portId: string) => boolean`. If the interceptor returns `true`, the provider immediately halts propagation, dropping the message before it reaches the `handleIncomingMidi` pipeline.
* **Component Consolidation:** The provided external UI components (`TransformationsContainer`, `TransformationsToolbar`, etc.) will be placed in `src/components/toolbar/`.
* **Mathematical Modifications:** * Diatonic/Chromatic shifts must multiply their delta by the `stepSize` payload.
  * PCS Rotation must calculate `(currentPcsIndex + (delta * stepSize) + (pcs.length * stepSize)) % pcs.length`.
  * Velocity calculation for the `PLAY` momentary action must calculate: `Math.max(1, Math.floor(((rect.bottom - clientY) / rect.height) * 127))`.

### Phase 4: Pre-Flight MIDI Interceptor & Learn Mode — 🔘 Pending
* **Objective:** Architect the MIDI blocker to capture hardware events before they hit the notation engine.
* **Instructions:**

  2. Update the `onmidimessage` assignment in `MIDIProvider.tsx`'s `useEffect`:
     ```typescript
     input.onmidimessage = (event: MIDIMessageEvent) => {
       const portId = input.id;
       const data = event.data as Uint8Array;
       if (midiInterceptorRef.current && midiInterceptorRef.current(data, portId)) return; // BLOCKED
       handleIncomingMidi(data, false);
     };
     ```
  3. In `TransformationsContainer.tsx`, register an interceptor via `useMidi()`.
  4. The interceptor logic: Parse `status` and `note`. Create a key string: `[portId]_[channel]_[note]`. Compare against the active `configs`.
  5. If `learnState.isActive`, capture the exact `portId`, `channel`, and `note` and assign it to the current target button, advancing the sequence. Return `true` to block.
  6. If NOT learning, check if the incoming `[portId]_[channel]_[note]` matches a saved config. If it does, execute that button's action (`handleButtonDown`) and return `true` to block it from the ROMPLER.
