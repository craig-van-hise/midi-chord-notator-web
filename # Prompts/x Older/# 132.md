# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
Based on a deep architectural review of the provided scripts, the audio playback inconsistencies are caused by three specific architectural gaps:
1. **Autoplay Policy Violation:** `audioEngine.init()` is never explicitly invoked by a global user gesture. Tone.js is attempting to start when the first MIDI payload arrives or when the React tree renders, which Chrome blocks.
2. **Masked Buffer Errors:** In `NotationCanvas.tsx`, the `catch` blocks around `audioEngine.noteOn()` are masking the real error (which is `smplr` throwing because the `AudioContext` is suspended/uninitialized).
3. **Missing Mutation Event Plumbing:** The `applyChromaticShift`, `applyDiatonicShift`, and `applyPcsRotation` functions in `NotationCanvas.tsx` synchronously update the pitch state but completely lack audio playback hooks. The toolbar event listener (`APP_TRANSFORM`) conditionally attempts to play audio based on a fragile `isUiClick` boolean, completely bypassing keyboard shortcuts (`Alt+Up/Down`).

**Objective:** Implement a global initialization gesture to unlock the `AudioContext`, and centralize audio playback directly inside the pitch mutation functions so both hardware keyboard and UI toolbar actions deterministically trigger audio.

## 2. Technical Decisions & Dependencies
* **Global Initialization:** `App.tsx` will host a one-time global `pointerdown` / `keydown` listener to invoke `audioEngine.init()`.
* **Centralized Playback Plumbing:** The fragile `isUiClick` check in the `APP_TRANSFORM` listener will be deleted. Instead, the `applyChromaticShift`, `applyDiatonicShift`, and `applyPcsRotation` functions will inherently calculate the resulting string array and call the existing `playPreviewNotes()` function.
