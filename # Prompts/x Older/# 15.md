# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The previous attempt to prevent MIDI port loopbacks (selecting the same physical device as both Input and Output) failed. The worker agent relied solely on a reactive UI `disabled` attribute, which is insufficient. If an output is already selected, and the user selects the matching input, the state collides. 

We must implement an "Ironclad Loopback Prevention" architecture. This requires intercepting the state change at the Context/Handler level, forcing an automatic eviction of the Output port if a collision is detected, and rejecting any incoming state changes that attempt to create a loopback.

## 2. Technical Decisions & Dependencies
* **Target Files:** `src/midi/MidiPortSelector.tsx`, `src/midi/MIDIProvider.tsx`.
* **String-Based Validation:** Web MIDI IDs differ between Input and Output ports on the same device. All collision detection MUST rely on evaluating the `port.name` string.
* **Three-Tier Block:** 1. `onChange` early return (Reject invalid output selection).
  2. Cross-state eviction (Input selection automatically clears a matching Output).
  3. `<option>` UI disable.

## 3. Implementation Phases

### Phase 1: State-Level Eviction & Handler Blocking — ✅ Completed

       ```typescript
           return; // Abort the state update
       }
       ```
