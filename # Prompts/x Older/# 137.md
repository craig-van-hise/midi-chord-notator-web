# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
The previous implementations of the Audio Gatekeeper failed due to two fundamental architectural flaws:
1. **Trapped DOM Node:** The overlay was placed inside the `NotationCanvas` (a `relative`, `h-[320px]` container), causing it to be hidden or obscured by canvas elements.
2. **Instrument Loading Race Condition:** Because the audio context cannot be initialized (`audioEngine.init()`) until a user gesture occurs, any React components (like the Rompler footer) that attempt to call `loadInstrument()` on mount immediately abort because the engine is not ready. This leaves the engine permanently devoid of samples.

**Objective:** 1. Rip out the failed overlay from `NotationCanvas.tsx`.
2. Implement a `fixed`, full-screen Gatekeeper in `App.tsx` that sits at the absolute top level of the DOM.
3. Explicitly execute `audioEngine.init()` AND `audioEngine.loadInstrument('piano')` sequentially inside the unlock gesture to guarantee the engine is hydrated with sound.
4. Add native locking to `engine.ts` so we don't rely on React closures.

## 2. Technical Decisions & Dependencies
* **Top-Level DOM:** The overlay will be placed in `AppContent` (`App.tsx`) with `fixed inset-0 z-[999999]`.
* **Native Engine Locks:** `AudioEngine` will use an internal `isUnlocked` property. This is infinitely safer than checking React `useRef` inside closure-heavy `useEffect` blocks.
