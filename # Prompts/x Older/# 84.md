# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The UUID selection engine is active, but we have exposed two critical data-handling bugs during transpositions. 
1. **Keyboard Metadata Loss:** Mutating a note via transposition causes the virtual piano highlights to disappear. This indicates the mutation function is stripping required metadata (like `velocity` or `source`) from the `ActiveNoteData` object before sending it to the global state.
2. **Cohemitonia Phantom Duplication:** When transposing a note creates a minor 2nd interval (e.g., crossing F over to E), the Dual-Column collision engine accidentally clones the top note, rendering three notes instead of two. The layout algorithm must be patched to prevent duplicate UUIDs from entering the final render array.

## 2. Technical Decisions & Dependencies
* **Data Preservation (Spread Operator):** All transposition mutations MUST use the ES6 spread operator (`...note`) to ensure every single property (velocity, channel, source, etc.) survives the mutation.
* **UUID Deduplication:** The layout recalculation function (specifically the dual-column zippering logic for minor 2nds) must be audited. A note object (identified by its UUID) must only ever be assigned to one rendering column, never both.

### Phase 2: Patch the Cohemitonia Phantom Note — ✅ Completed
* **Objective:** Prevent the Dual-Column collision engine from cloning notes at minor 2nd intervals.
* **Instructions:**
    1. Open `src/components/NotationCanvas.tsx` or `src/utils/notationMath.ts` and locate the layout rendering loop (where notes are separated into `left` and `right` columns to avoid collisions).

    3. The bug occurs because the loop likely evaluates the current note against the previous note, and accidentally pushes one of them into a render array twice.
