# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
The objective is to implement a robust Note Selection and Editing Engine into the `NotationCanvas` for a single vertical sonority. Previous attempts failed critically due to architectural over-engineering. We are starting from a stable, rolled-back state. The core mandate is to avoid complex canvas math in favor of DOM-based collision/delegation, completely bypass the font/asset pipeline, and maintain the imperative rendering paradigm to prevent React lifecycle race conditions. Crucially, pitch set rotation must integrate directly with the theoretical `PCS_LUT` engine.

## 2. Technical Decisions & Dependencies

* **STRICT PROHIBITION - Asset Pipeline:** You are absolutely forbidden from altering `vite.config.ts`, changing CSS `url()` paths in `index.css`, or moving `Bravura.woff2`. The SMuFL font resolution is currently stable.
* **Hit-Testing Architecture (DOM Delegation):** Use standard DOM Event Delegation (`e.target.closest('[data-note-id]')`). This guarantees perfect hit-testing with zero coordinate fragmentation.
* **Marquee Architecture (AABB):** For drag-to-select, render an absolute-positioned `<div>` overlay. Calculate intersection using standard Axis-Aligned Bounding Box (AABB) math against the `getBoundingClientRect()` of the DOM note elements.
* **Selection State Mutability:** Selection state MUST be managed via a mutable Ref (e.g., `selectedNoteIds.current = new Set<string>()`) and visual updates MUST be applied imperatively (e.g., toggling a `data-selected="true"` attribute).
* **Theoretical PCS Rotation:** Chord inversion must not be handled by naive octave displacement. It must extract the Pitch Class Set (PCS), perform array rotation (e.g., `[0,4,7]` -> `[4,7,0]`), and query `binaryLut.ts` to reconstruct the musically accurate sonority.

### Phase 3: Keyboard Navigation & Theoretical PCS Rotation — ✅ Completed
* **Objective:** Implement keyboard-driven selection traversal and theoretically accurate pitch mutation via the LUT.
* **Instructions:**
    1.  Attach a `keydown` listener to the document or canvas container.
    2.  **Navigation (Up/Down Arrow):** If a single note is selected, find its index. Pressing Up selects the `index + 1` note; Down selects `index - 1`.
    3.  **Transposition Mutations:** * `Shift + Up/Down`: Increment/Decrement target note's absolute `midiNote` by 1.
        * `Cmd/Ctrl + Up/Down`: Increment/Decrement by 12.
        * `Option + Cmd + Up/Down (PCS Rotation)`: 
            * Extract the Pitch Class Set of the currently selected notes (e.g., map `note % 12` and remove duplicates to form the PCS array).
            * Perform mathematical array rotation. If UP: shift the first element and push it to the end (e.g., `[0, 4, 7]` becomes `[4, 7, 0]`). If DOWN: pop the last element and unshift it to the front.
            * Query `binaryLut.ts` / `chordSpeller.ts` with the newly rotated PCS to retrieve the structural and spelling metadata.
            * Reconstruct the absolute MIDI pitches based on the rotated intervals starting from the closest logical bass note, ensuring the new voicing matches the LUT's rotational data.
    4.  **Re-render Trigger:** After mutating `activeNotes.current`, manually re-trigger the imperative layout function (`updateSpellings()` -> `recalculateLayout()`).
