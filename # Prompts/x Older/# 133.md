# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The application is experiencing two critical audio playback regressions:
1. **Autoplay Policy Block on MIDI Input:** Web MIDI events do not qualify as "user gestures" under modern browser Autoplay policies. If a user loads the page and immediately plays a hardware MIDI keyboard without clicking the DOM first, the `AudioContext` remains suspended and silent.
2. **Bypassed Audio in Keyboard Shortcuts:** The `handleKeyDown` event listener in `NotationCanvas.tsx` calls transformation functions (`applyChromaticShift`, `applyDiatonicShift`, `applyPcsRotation`) directly. It currently lacks the instruction to play the audio engine after these mutations occur, whereas the UI toolbar correctly dispatches an event that triggers playback.

**Objective:** Plumb audio playback explicitly into the keyboard shortcut event handler and ensure the `AudioContext` is unlocked on the very first physical DOM interaction.

## 2. Technical Decisions & Dependencies
* **DOM Gesture Unlocking:** We must attach a one-time, global `pointerdown` and `keydown` listener to the `window` to silently execute `audioEngine.init()` and unlock Tone.js the moment the user touches the app interface.
* **Direct Shortcut Plumbing:** We will extract the array-mapping logic that converts `selectedNoteIds` to string arrays and inject it directly into the `handleKeyDown` block in `NotationCanvas.tsx`.

* **Instructions:**

       ```typescript
         .map(id => activeNotes.current.find(n => n.id === id)?.note)
         .filter((n): n is number => typeof n === 'number')
         .map(pitch => Tone.Frequency(pitch, "midi").toNote());
       
       if (transposedStrings.length > 0) {
         playPreviewNotes(transposedStrings, true);
       }
       ```
