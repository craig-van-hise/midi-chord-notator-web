# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
The application successfully detects and selects MIDI Output ports, but it is currently not transmitting any MIDI data to them. To fix this, we need to implement a "MIDI Thru" architecture. 

When a user plays their controller (or clicks the virtual UI keyboard), the incoming raw `Uint8Array` data needs to be immediately forwarded to the `selectedOutputPort` via its native `.send()` method. This must happen *before* our internal sustain pedal logic intercepts and modifies the note-off flow, ensuring that external hardware synthesizers receive the exact raw data (physical Note Offs and CC 64 events) to manage their own polyphony natively.

## 2. Technical Decisions & Dependencies
* **Stale Closure Prevention:** Because `handleIncomingMidi` is a high-frequency callback, adding `selectedOutputPort` to its dependency array would cause constant function recreation and potential React lifecycle lag. We will use a `useRef` (`selectedOutputPortRef`) to hold the current output port, allowing $O(1)$ access inside the hot loop without dependency churn.
* **Architecture Positioning:** The `.send()` call must be placed at the absolute top of `handleIncomingMidi`. This guarantees that both physical MIDI events (from the hardware controller) and virtual MIDI events (from the on-screen keyboard) are forwarded to external gear *before* the UI-specific sustain math intercepts them.
