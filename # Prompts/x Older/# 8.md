# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
The previous phase executed a rewrite of `src/utils/notationMath.ts` to support diatonic scale generation and octave boundary correction. While the isolated unit tests passed, the implementation has completely failed in the browser environment. The UI continues to render Cb as B. 

This indicates a critical integration failure. The `keySignature` value being passed from `MIDIProvider` to `NotationCanvas` to `getEnharmonicSpelling` is either the wrong type (e.g., an integer instead of a string), undefined, or failing the dictionary lookup in `KEY_SIG_ACCIDENTALS`, causing the engine to silently swallow the error and default to a C Major chromatic fallback.

We must immediately audit the data bridge, strictly type the key signature payload, and force the mathematical engine to fail loudly if given bad data.

## 2. Technical Decisions & Dependencies
* **Files to Modify:** `src/midi/MIDIProvider.tsx`, `src/utils/notationMath.ts`, `src/components/NotationCanvas.tsx`.
* **Type Strictness:** The application must agree on a single source of truth for the `keySignature` type. If it is a string (e.g., `"Gb Major"`), the Provider, Selector, and Math Engine must all explicitly enforce this `Type`.
* **Fail-Loud Architecture:** `getDiatonicMap` must throw an explicit error if the provided `keySignature` is not found in the dictionary, rather than silently failing. This will expose exactly where the data is malformed.
