# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
This sprint addresses a complex state-mutation bug within the Voicing-Aware PCS Rotation engine.

**The Bug:**
When explicitly spelled chords (e.g., F# Major) are rotated, the notation, chord symbol, and spelled note labels become scrambled. This occurs because the `spellingOverride` string is statically tied to the `ActiveNoteData` object ID. When the rotation logic shifts the mathematical pitch of the object to the next chord tone, the object retains its original spelling override (e.g., the note moving to A# inherits the "F#" spelling).

**The Solution:**
Overrides must be mapped by Pitch Class (0-11) during rotations. Before shifting pitches, the rotation block must extract a dictionary of the current overrides mapped to their pitch classes. When mapping the new target pitch to the object, the object must inherit the override associated with its *new* target pitch class. Additionally, `getChordSymbol` must be hardened to ensure bass-note inversions respect these overrides.

## 2. Technical Decisions & Dependencies
* **Pitch Class Mapping:** The rotation mapping function in `NotationCanvas.tsx` will utilize a temporary `pcOverrides` record to safely swap the overrides between the mutating objects.
* **Chord Symbol Hardening:** The `getChordSymbol` function in `chord_speller.ts` must explicitly check the `overrides` dictionary when formulating the string for the bass note in slash chords.
