# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The application is correctly generating double flat (`bb`) and double sharp (`x`) root names for the primary chord symbol (e.g., `Bbb` and `Fx°7`), but the individual note spellings generated by `getChordSpelling` are failing to inherit these double accidentals. Instead, they are reverting to single accidentals or naturals (e.g., spelling a Bbb major chord as `Bb, D, F`).

The root cause lies in `getIntervalBetweenPitches`. This function calculates the relative semitone distance between the key center and the chord root by referencing the `PITCH_TO_PC` dictionary. Because double accidentals like `"Bbb"` and `"Fx"` are missing from this dictionary, the fallback logic evaluates them by truncating the string (e.g., `"Bbb"` becomes `"Bb"` [PC 10 instead of 9], and `"Fx"` becomes `"F"` [PC 5 instead of 7]). This incorrect pitch class cascades through the interval summation engine, offsetting the entire chord spelling.

## 2. Technical Decisions & Dependencies
* **Dictionary Expansion:** We must explicitly add the full theoretical spectrum of double flats and double sharps to the `PITCH_TO_PC` dictionary.
* **No Side Effects:** This is purely a data mapping fix. The interval math in `sumIntervalStrings` and `convertIntervalToPitchSpelling` already supports `bb` and `x` offsets naturally; they merely lacked the accurate starting pitch class to build from.
