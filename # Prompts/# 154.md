# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
**Context:** The Grand Staff MIDI Notator requires dynamic enharmonic spelling for symmetrical pitch class sets (Tritones, Diminished 7ths, Augmented Triads). A previous attempt to implement this failed at the integration layer. The logic utility `symmetricalSpeller.ts` was built, and `chordSpeller.ts` was modified, but the `keyCenterPc` (derived from the active key signature) and `lowNotePc` (derived from live MIDI input) were not properly plumbed from the `NotationCanvas.tsx` / `MIDIProvider.tsx` layer down into the spelling interceptor. As a result, the live application continues to use static spellings from the PCS LUT.
**Objective:** Forensically repair the data pipeline in `NotationCanvas.tsx` and `chordSpeller.ts`. We must ensure that whenever `updateSpellings` runs in the canvas, it correctly calculates the `keyCenterPc` from the current `keySignatureRef.current`, extracts the lowest pitch from the currently active notes, and passes these variables accurately into `getChordSpelling` and `getChordSymbol`. Crucially, we must ensure the overridden rotation arrays from `symmetricalSpeller.ts` are actually mapped to the final SMuFL glyphs rendered on the canvas.

## 2. Technical Decisions & Dependencies
* **Data Flow Trace:** 1. In `NotationCanvas.tsx`, `updateSpellings()` relies on `getChordSpelling()` and `getChordSymbol()`.
    2. Looking at the provided `NotationCanvas.tsx` (lines 538-550), it currently calculates `keyCenterPc` and passes it. 
    3. *However*, looking closely at the provided `chordSpeller.ts` (lines 280-305), when `symIntervals` is returned from `getSymmetricalSpelling`, the code does: `intervals = symIntervals;`. 
    4. **THE BUG:** The `symIntervals` (e.g., `["b5", "1"]`) are standard degree intervals, but the subsequent mapping block (lines 310-335) in `getChordSpelling` attempts to match `semitones` against `intervals` using `majorSteps`. Because `symIntervals` are just strings in an array, `toneInterval = intervals[idx];` assigns the interval correctly, BUT the root interval calculation (`rootRelKeyInterval`) and absolute interval calculation (`sumIntervalStrings`) might be yielding incorrect results or falling back to the individual note speller (lines 337-346).
    5. **Correction Strategy:** We need to ensure that when `symIntervals` is active, the logic perfectly bypasses the static LUT interval parsing and directly uses the dynamic `symIntervals` to calculate the final `absoluteInterval` relative to the *Global Key Center*, not just the relative root.
