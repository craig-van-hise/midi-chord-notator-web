# Product Requirements Prompt (PRP) — Remediation Patch 🔘

## 1. Project Context & Objectives
**Project:** Grand Staff MIDI Notator
**Current State:** The `applyGlobalOctaveWrap` math utility is causing "voicing compaction." When chords spanning more than one octave hit the 21 or 108 bounds, the out-of-bounds notes wrap around into the same octave as the other notes, permanently destroying open voicings and deleting duplicate pitch classes. 
**Objective:** Abandon the "Wrap" strategy entirely in favor of a **Strict Boundary Block**. If a mathematical transformation pushes *any* note within a chord out of the 21-108 range, the entire transformation must be aborted, preserving the chord's original voicing exactly where it was.

## 2. Implementation Phases

### Phase 1: Math Utility Overhaul — 🔘 Pending
* **Objective:** Replace the wrapping logic with a strict validation gatekeeper.
* **Instructions:**
  1. Open the file containing `applyGlobalOctaveWrap` (e.g., `src/utils/notationMath.ts`).
  2. Delete `applyGlobalOctaveWrap` and replace it with a new function `enforcePianoRange`.
  3. The new function must compare the `proposedNotes` against the `originalNotes`.
     ```typescript
     export const enforcePianoRange = (proposedNotes: number[], originalNotes: number[]): number[] => {
       // Check if ANY note in the proposed transformation falls off the 88-key piano
       const isOutOfBounds = proposedNotes.some(n => n < 21 || n > 108);
       
       if (isOutOfBounds) {
         // If transforming an existing chord, reject the shift entirely to preserve voicing
         if (originalNotes.length > 0) {
           return originalNotes;
         }
         // Fallback for fresh live inputs: just strip the impossible notes
         return proposedNotes.filter(n => n >= 21 && n <= 108);
       }
       
       return proposedNotes;
     };
     ```
* **TDD Checkpoint:**
    * **Test Case 1 (Valid Shift):** Given `proposed = [60, 64, 67]`, `original = [48, 52, 55]`. When enforced, Then returns `[60, 64, 67]`.
    * **Test Case 2 (Out of Bounds Transformation Block):** Given `proposed = [105, 109]`, `original = [93, 97]`. When enforced, Then returns `[93, 97]` *(Because 109 > 108, the whole shift is rejected)*.
    * **Test Case 3 (Deep Out of Bounds Block):** Given `proposed = [8, 12, 15]`, `original = [20, 24, 27]`. When enforced, Then returns `[20, 24, 27]` *(Because 8 and 12 < 21)*.
    * **Test Case 4 (Fresh Input Fallback):** Given `proposed = [20, 60]`, `original = []`. When enforced, Then returns `[60]`.
