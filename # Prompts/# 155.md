# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
**Context:** The Symmetrical Spelling interception pipeline is failing in the live production environment. The application uses a mathematically derived decimal bitmask to index a binary LUT (`PCS_LUT.dat`). For perfectly symmetrical chords, normalization relative to the bass note yields the exact same decimal index regardless of inversion (e.g., Tritones always equal decimal `65`). 
**Problem:** The previous implementation attempted to intercept spellings by checking `entry.pitch_class_set`. Because this array is stripped from the optimized binary `.dat` file at runtime, the interceptor receives `undefined`, fails silently, and falls back to the static, unrotated LUT spelling.
**Objective:** Refactor the symmetrical interceptor to rely exclusively on the `decimal` bitmask (`65`, `273`, `585`) rather than relying on LUT payload arrays.

## 2. Technical Decisions & Dependencies
* **Bitmask Identification:** * Tritone (`[0, 6]`): `2^0 + 2^6` = **65**
    * Augmented Triad (`[0, 4, 8]`): `2^0 + 2^4 + 2^8` = **273**
    * Diminished 7th (`[0, 3, 6, 9]`): `2^0 + 2^3 + 2^6 + 2^9` = **585**
* **Architecture:** We will modify `symmetricalSpeller.ts` to accept the `decimal: number` instead of `pcsPrime: number[]`. This completely decouples the dynamic spelling logic from the internal structure of the LUT entries.
