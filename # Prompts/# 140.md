# Product Requirements Prompt (PRP) — 🔵 Active

## 1. Project Context & Objectives
**Project:** Grand Staff MIDI Notator
**Current State:** The system has playable MIDI transformations (velocity/duration passthrough) and a working LISTEN mode.
**Objective:** Streamline the navigation UI by removing deprecated Undo/Redo functionality, completely overhaul the "Home" button to track and revert to the foundational (pre-transformation) input chord, and introduce a global "VELOCITY" parameter for UI-triggered transformations.

## 2. Technical Decisions & Dependencies
* **State Management:** Add `homeChord` (array of active notes) and `uiVelocity` (number, 0-127, default 80) to the global/persistent state configuration.
* **Home Chord Anchor Logic:** The `homeChord` state serves as the anchor point. It MUST ONLY be overwritten when a user inputs a *new* chord natively via hardware (MIDI controller) or the on-screen UI Keyboard (specifically in toggle mode). Transformations (transpositions, rotations) manipulate the active display chord but MUST NOT overwrite the `homeChord`.
* **Home Playback Mechanics:** Pressing the Home button executes a reversion to the `homeChord`. It must instantly update the visual canvas and trigger an audio preview using the identical duration and audio-bypass ("LISTEN" mode) logic as the standard Play button.
* **UI Velocity Parameter:** Since clicking a UI Transformation button lacks physical velocity data, the new `uiVelocity` parameter dictates the dynamic level passed to Tone.js for on-screen clicks. (Note: hardware MIDI triggers will continue to use their native physical velocity, as implemented previously).

## 3. Implementation Phases

  4. **Apply UI Velocity:** Update the onClick handlers for UI transformation buttons (e.g., in `TransformationsToolbar.tsx`) to pass the global `uiVelocity` state to the audio engine's attack trigger, replacing any hardcoded values.
