# Product Requirements Prompt (PRP) — 🔵 Active

## 1. Project Context & Objectives
A critical z-index stacking context bug has been identified. Modal overlays (like the Info Modal and Settings Modal) are being visually pierced by the `KeySignatureSelector` dropdown. 

This occurs because the Modals are currently being rendered inside the `<header>` element, trapping them within the header's localized stacking context. To ensure modals can obscure the entire viewport globally, their component calls must be hoisted to the absolute root level of the application's DOM tree.

## 2. Technical Decisions & Dependencies
* **Files to Modify:** `src/App.tsx`, `src/components/SettingsModal.tsx` (or equivalent), `src/components/InfoModal.tsx` (or equivalent).
* **Stacking Context Override:** Modal components must be rendered as the *last direct children* of the main top-level wrapper in `App.tsx`, completely outside of the `<header>`, `<main>`, or `<footer>` elements.
* **Ironclad Z-Index:** The modal overlay backdrop must be explicitly set to `z-[100]` to guarantee it sits above all UI layers, including the absolute-positioned notation elements.

## 3. Implementation Phases

### Phase 1: Modal Component Hoisting (Re-parenting) — ✅ Completed
* **Objective:** Move the modal rendering logic to the root application level.
* **Instructions (Target: `src/App.tsx`):**
    1. Locate the state variables controlling modal visibility (e.g., `isInfoModalOpen`, `isSettingsModalOpen`). *Ensure these remain in `App.tsx`.*
    2. Locate the actual component rendering calls (e.g., `{isInfoModalOpen && <InfoModal ... />}`) currently sitting inside the `<header>` block alongside the buttons.
    3. **Remove** the component calls from the `<header>`. (Leave the trigger `<button>` elements in the header).
    4. **Paste** the component calls at the very bottom of the JSX return statement, inside the top-level `<div className="flex flex-col h-screen...">` wrapper, immediately after the `<footer>` block.
