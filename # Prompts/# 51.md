# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
The application is currently experiencing infinite MIDI feedback loops when routed through virtual loopback drivers (like the macOS IAC Driver). 

The root cause is an architectural flaw in `src/midi/midiAccess.ts`. The `requestMidiAccess` initialization function blindly attaches an `onmidimessage` event listener to `midiAccess.inputs.forEach(...)`. This means the application is globally listening to all available MIDI inputs simultaneously, rather than strictly obeying the user's `selectedInputPort`. When data is sent out to a loopback driver, it arrives at the loopback input, gets intercepted by our global listener, and is sent right back out.

We must dismantle the global event bridge (`RAW_MIDI_MESSAGE_RECEIVED`) and refactor `MIDIProvider.tsx` to explicitly attach and detach its `onmidimessage` listener strictly to the `selectedInputPort` managed in state.

## 2. Technical Decisions & Dependencies
* **Targeted Listeners:** The `onmidimessage` callback will be moved entirely into a `useEffect` inside `MIDIProvider.tsx` that strictly depends on `selectedInputPort`. 
* **Garbage Collection:** When the `selectedInputPort` changes, the cleanup function of the `useEffect` MUST set the previous port's `onmidimessage` to `null` to prevent ghost listeners.
* **Dead Code Elimination:** The `RAW_MIDI_MESSAGE_RECEIVED` CustomEvent bridge is no longer necessary for incoming hardware MIDI, as the Provider will handle the native event directly.
