# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
This sprint executes a major simplification of the MIDI routing engine and consolidates the core UX interaction modes.

**Objective 1: The OMNI Matrix & Output Eradication**
Remove the concept of MIDI Output entirely. Introduce an "OMNI (All Ports)" input mode as the default, dynamically binding the `onmidimessage` listener to an array of all active MIDI inputs.

**Objective 2: Mode Consolidation & Migration**
The "Toggle Mode" and "Hold Mode" buttons are currently residing in `Keyboard.tsx`. We must move them into `SettingsModal.tsx` as persistent settings. Both modes must be defaulted to `true`. Furthermore, `NotationCanvas.tsx` must be rewritten so that both modes can operate simultaneously (Toggle behavior overrides the "chord flush" of Hold mode, but both prevent note deletion on key release).

## 2. Technical Decisions & Dependencies
* **Global Context State:** You will add `isToggleModeActive` to the `MidiContextType` in `MIDIProvider.tsx` (alongside the existing `isHoldModeActive`). Both must initialize to `true`.
* **OMNI Listener Engine:** The `onmidimessage` attachment in `MIDIProvider.tsx` will dynamically map to `Array.from(midiAccess.inputs.values())` when the selected input ID is `'omni'`.
* **Keyboard Simplification:** Because the `NotationCanvas` computational engine handles mode logic, `Keyboard.tsx` will be stripped of its local `isToggleMode` state. Virtual key clicks will simply emit raw Note On / Note Off events, letting the canvas intercept and process them based on the active global modes.
