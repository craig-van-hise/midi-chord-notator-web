# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
This sprint addresses two severe UX bugs introduced during the Spelling Lock implementation.

**The Bugs:**
1. **Transposition Freeze:** Notes written via Write Mode cannot be transposed visually, and diatonic transposition logic breaks after one iteration. This occurs because the `isLockedSpelling: true` flag prevents `updateSpellings()` from recalculating the note's `stepOffset` after the pitch has been mathematically altered by the keyboard handlers.
2. **Toolbar Bubbling:** Clicking the Accidental Override toolbar exits Write Mode because the pointer events bubble up to the `NotationCanvas` container, which interprets them as "click-away" deselection events.

## 2. Technical Decisions & Dependencies
* **Spelling Lock Release:** The `isLockedSpelling` flag represents an explicit, absolute static pitch declaration. If the user selects that note and mathematically transposes it (via `ArrowUp`/`ArrowDown`), the note is fundamentally changing context. Therefore, all transposition operations must explicitly set `isLockedSpelling: false` on the mutated note objects, allowing `updateSpellings()` to resume rendering their new positions.
* **Event Propagation:** The toolbar container must explicitly call `e.stopPropagation()` on all pointer events to shield itself from the canvas's hit-testing layer.
