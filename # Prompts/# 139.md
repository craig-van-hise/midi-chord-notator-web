# Product Requirements Prompt (PRP) — 🔵 Active

## 1. Project Context & Objectives
**Project:** Grand Staff MIDI Notator
**Current State:** The system has a stable core audio gatekeeper, a centralized settings modal, and direct audio transformation plumbing. 
**Objective:** Execute a UI and logic overhaul of the MIDI Learn and Transformation tracking systems. Specifically, we need to introduce full CRUD capabilities to MIDI mappings (Clear All, Clear specific, manual edit), wire up a global "LISTEN" mode toggle for silent transformations, and completely overhaul the transformation playback engine so that mapped MIDI triggers pass their dynamic velocity and duration (Note On/Off states) through to the Tone.js audio engine, making transformations fully playable.

## 2. Technical Decisions & Dependencies
* **State Management:** All MIDI mapping state MUST continue utilizing the existing `usePersistentState.ts` hook. Do not introduce Redux or Zustand. 
* **React 19 & Tailwind 4.1 Mandate:** Continue using functional components exclusively. Rely on the React Compiler for memoization (no `useMemo`/`useCallback`). UI elements must use Tailwind 4.1 utility classes via `@import "tailwindcss"`.
* **Playable Transformation Architecture:**
  * To support dynamic durations, the system can no longer use fixed-duration `triggerAttackRelease` calls for MIDI-triggered transformations.
  * The `engine.ts` must expose independent `triggerAttack(notes, velocity)` and `triggerRelease(notes)` methods.
  * The `MIDIProvider.tsx` (or the central MIDI message handler) must maintain an in-memory `Map<number, number[]>` bridging the incoming hardware `triggerNote` to the resulting `transformedChordNotes`. 
  * When a `Note Off` (or Note On with 0 velocity) is received for a mapped `triggerNote`, the router must look up the playing chord notes and execute a `triggerRelease` on them to stop the audio.
* **Listen Mode Logic:** The `listenMode` state dictates audio rendering *only*. When disabled, visual notation transformations must still execute flawlessly, but the call to `engine.ts` must be bypassed.

### Phase 3: Playable Transformations (Velocity & Duration Passthrough) — ✅ Completed
* **Objective:** Refactor the MIDI router and audio engine to track mapped Note On/Off states, passing velocity and matching release commands to make transformations "playable".
* **Instructions:**
  1. **Engine Updates:** Verify or update `src/audio/engine.ts` to expose `triggerAttack(notes: number[], velocity: number)` and `triggerRelease(notes: number[])`.
  2. **Router State:** In `src/midi/MIDIProvider.tsx` (or the respective MIDI message handler), instantiate a `Map<number, number[]>` named `activeTransformationNotes`. This will map a physical trigger Note Number to the array of MIDI note numbers generated by the transformation.
  3. **Note On Interception:** When a MIDI message matches a mapped transformation:
     * Calculate the new transformed chord.
     * Extract the raw velocity from the incoming `MIDIMessageEvent` payload (data2) and normalize it (0-1) if required by the audio engine.
     * Store the resulting chord array in `activeTransformationNotes` using the trigger note number as the key.
     * Call `audioEngine.triggerAttack(transformedChord, velocity)`.
  4. **Note Off Interception:** When a `Note Off` (or `Note On` with velocity 0) arrives for a mapped trigger note:
     * Check if the note exists in `activeTransformationNotes`.
     * If found, retrieve the array of transformed notes.
     * Call `audioEngine.triggerRelease(transformedNotes)`.
     * Delete the entry from the Map.
