# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The application is suffering from a **Hardware-State Desync** (Ghost Notes) during computational transpositions. 

**The Bug:** When a user computationally transposes a note (e.g., via `Alt + ArrowUp`), the underlying `noteData.note` property is modified. However, the physical MIDI hardware (or virtual keyboard) is still tracking the *original* MIDI pitch. If the physical key sends repeated `Note On` data (via active sensing, aftertouch, or accidental re-trigger), the `NotationCanvas` fails to recognize it as an existing note, pushing a duplicate into the `activeNotes` array. Furthermore, when the user lifts the physical key, the `Note Off` event fails to find and splice the note because its `note` value no longer matches the physical pitch. This results in "detritus" notes (like the unremoved `Bb` and `D` in the provided screenshot) merging with the newly transposed notes.

**The Solution:** We must permanently bind the hardware's physical pitch to the active note object by introducing an immutable `sourceMidi` property. The `Note On` collision detection and `Note Off` garbage collection must evaluate both the current computational `note` and the original `sourceMidi`.

## 2. Technical Decisions & Dependencies
* **Pipeline Hardening:** You are strictly forbidden from modifying the transposition math. You will exclusively modify the state hydration and reconciliation logic within `NotationCanvas.tsx`.
* **Immutability:** The new `sourceMidi` property must remain untouched during all transpositions, acting as the absolute source of truth for physical hardware reconciliation.
