# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The application is experiencing a **One-Way State Desync** between the computational engine (`NotationCanvas`) and the global state provider (`MIDIProvider`). 

**The Bug:** When transposed notes are cleared via UI interactions (such as turning off "Toggle Mode", triggering a Panic, or even a standard Note Off event after a transposition), `NotationCanvas` successfully clears its local `activeNotes.current` array and removes the notes from the SMuFL rendering. However, it fails to broadcast this clearance back to the `MIDIProvider`. Because the `Keyboard` component derives its active highlights from `MIDIProvider`, it retains the "ghost notes" indefinitely.

**The Solution:** We must enforce a strict two-way data binding protocol. Whenever `NotationCanvas` mutates its local `activeNotes.current` array (additions, deletions, or complete resets), it must push the updated state to `updateActiveNotes` so the rest of the application (like the `Keyboard`) stays synchronized.

## 2. Technical Decisions & Dependencies
* **Data Flow Mandate:** `NotationCanvas` currently acts as the absolute source of truth for mutated pitch data. You must ensure that every execution branch that removes or clears a note from `activeNotes.current` also calls `updateActiveNotes([...activeNotes.current])`. 
* **Reference Equality:** Always spread or clone the array (`[...activeNotes.current]`) when calling `updateActiveNotes` to ensure React's reconciliation engine detects the state change and triggers a re-render on the `Keyboard`.
