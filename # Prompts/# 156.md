# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
**Context:** The `KeySignatureSelector` currently uses a single dropdown for Major keys. We are splitting this into a dual-dropdown architecture (Root + Scale Type). 
**Problem:** The current diatonic transposition engine (`KEY_UP`/`KEY_DOWN`) relies on fixed modulo-7 arithmetic, assuming 7-note scales where every pitch maps to adjacent staff lines. This breaks for variable-cardinality scales (like 5-note Pentatonics or 8-note Diminished scales) which require skipping staff lines to render correctly on the SMuFL canvas.
**Objective:** Decouple the Root and Scale Type in the UI. Overhaul the diatonic math engine to perform an instant O(1) lookup in the `PCS_LUT` using a pre-calculated decimal bitmask hash. It will parse the exact `scale_intervals` from that entry to mathematically derive the accurate SMuFL Y-coordinates (staff steps) directly from the interval strings.

## 2. Technical Decisions & Dependencies
* **Pre-calculated Decimal Hash:** The agent will use a hardcoded `SCALE_DECIMAL_MAP` to instantly access the exact row in the LUT without any O(N) array searching.
* **Interval Parsing Math:** The staff layout is strictly derived from the LUT's interval strings. 
  * Example: `"b3"` -> The regex extracts degree `3`. The staff step offset relative to the root is `3 - 1 = 2`.
* **Circle of Fifths Ordering:** The root dropdown must strictly follow the provided C-centered circle of fifths array.
